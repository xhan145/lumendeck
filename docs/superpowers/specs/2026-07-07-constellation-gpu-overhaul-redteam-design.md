# LumenDeck — Constellation GPU Visual Overhaul: Redteam Review + Staged Plan (2026-07-07)

**Status:** Redteam design review — pre-implementation gate.
**Scope reviewed:** the proposed GPU overhaul of the Living Constellation (Graph3DView): GPU-instanced stars, gravity field texture, spacetime-fabric mesh, ripple/wave propagation, GPU particles, bloom/glow post-processing, lens distortion, trails/wakes, 8-variable visual encodings, 4 quality tiers, FPS-based dynamic scaling, reduced-motion mode, hidden-tab throttling, GPU fallbacks.
**Evidence basis:** full read of `src/components/graph/Graph3DView.tsx` (1,716 lines) + `graph3d/*`, `src/state/*` (store.ts ~114KB), `src/core/field|motion|audio/*`, all constellation specs in `docs/superpowers/specs/`, and a repo-wide perf/platform sweep. All file:line claims below were verified against the working tree on 2026-07-07 (v0.19.1, branch `feature/creative-os`).

---

## Executive Verdict

**Staged rollout of a heavily cut-down subset. Do not ship as one overhaul. Do not build the proposal as written.**

The proposal imports assumptions from a different genre. It is designed for a 100,000-star astronomy scatterplot; the Living Constellation is a **10–60 node workflow editor** (default workflow: 10 nodes, `core/workflow.ts:159-189`; no node cap exists, but nodes are added one at a time from a palette) that shares one GPU with a **local diffusion renderer** — including the just-spec'd streaming preview that fires real sd-turbo renders from ghost drags (debounced ~150ms settle, ≤1 in flight, off by default — but it is the feature's intended headline use; `docs/superpowers/specs/2026-07-07-field-presets-streaming-preview-design.md:51-68`). At this scale, half the proposed systems solve problems the app does not have:

- **GPU-instanced stars** save ~59 draw calls at 60 nodes — unmeasurable — while breaking the per-orb ShaderMaterial tint, the interactive ring dial, CSS3D chip anchoring, raycast picking, and the "Editing never breaks" invariant. Negative value.
- **GPU particle systems + gravity field texture + compute-style simulation** are machinery for counts this app will never render. A few thousand CPU-simmed points cover every visible outcome.
- **FPS-based dynamic quality scaling** has nothing to scale against — there is no frame instrumentation in the codebase at all — and continuous auto-scaling oscillates. Static tiers plus two specific automatic downshifts are enough.
- **Lens distortion is not merely inadvisable — it is architecturally broken here.** Node cards and chips live in a CSS3DRenderer DOM layer that a WebGL post-pass cannot warp (`Graph3DView.tsx:437-438`; architecture decision in `2026-07-06-graph3d-and-image-node-design.md:10-31`). Any screen-space UV distortion desynchronizes the WebGL orbs from their DOM chips and from raycast picking. Cut permanently.

Meanwhile the proposal contradicts four documented, shipped decisions: the explicit **"No postprocessing passes (no bloom pipeline)"** decision with its 60fps@40-nodes budget (`2026-07-06-graph3d-and-image-node-design.md:62-63`), the **"the editor idles (dirty-flag) when static"** two-loop invariant (`2026-07-06-living-constellation-vision.md:59-61`, enforced by `flushScheduler.ts` and regression-tested), the **≤16MB MSI** acceptance gate, and the **"never color alone"** accessibility bar (`2026-07-02-lumendeck-design.md:71-73`). A plan that silently reverses standing decisions is how codebases lose their invariants; each reversal below is called out explicitly and either avoided or made a measured, flagged, default-off Phase 5 question.

**What survives, and is genuinely good:** the gravity fabric — *if built stateless*. Mass already exists in the data model (`orbWeight.ts` `primaryWeight`/`weightT`, the one number that best characterizes each node), and a fabric of depressions with equipotential contour lines would make the constellation's core metaphor legible at a glance. Event-driven ripples (a render completes → one ping), data-driven emissive glow, and trails tied to the motion/ghost systems also survive — because each maps to a real product datum. Everything else is cut, simplified, or deferred behind measurements.

The staged plan (Phases 0–5 below) starts with data plumbing and instrumentation — the two things the proposal forgot and the codebase lacks — and delivers the fabric as the first visible slice, feature-flagged and default-off.

## Top 10 Risks

### R1 — Always-on effects break the "provably sleeps" idle invariant — **Critical**
- **Why it matters:** The dirty-flag scheduler is a documented product invariant ("Two loops, cleanly separated: the editor idles (dirty-flag) when static", `living-constellation-vision.md:59-61`; "no idle animation burn", `graph3d-and-image-node-design.md:34`), implemented in `flushScheduler.ts:42-76` and protected by regression tests (`orbNodes.test.ts:192-249`). This is a local-first desktop app whose GPU also runs diffusion; idle burn is a real cost.
- **How it fails:** Ripples, particles, trails, and an animated fabric all want a continuous rAF. The naive implementation adds an unconditional loop, and the editor never sleeps again — battery drain, fan noise, and stolen GPU headroom on every idle window.
- **Mitigation:** All new effects draw only inside existing active states (playback, audio, pointer interaction) plus a short **settle clock**, armed only by decay-bearing events (a ripple ping, a well transition) — never by ordinary invalidations: animate ≤1.5s of decay, render one final settled frame, then sleep. While a continuous driver (playback/audio) owns the frame, the settle clock is suspended — decay animates for free on that loop and must not schedule extra renders (see the frame-hook contract in Rendering Architecture). The fabric must look correct as a *static image* (depressions + contours carry the data; motion is garnish). Add unit-tested guarantees, in the style of the existing flushScheduler tests, that the settle clock reaches sleep and that exactly one render happens per frame when playback and a decaying ripple overlap.

### R2 — GPU contention with diffusion renders during the exact gesture that matters — **Critical**
- **Why it matters:** The newest spec fires 320px/4-step sd-turbo previews from ghost/orb drags with a "feels near-live on 8 GB" target and <1.5s acceptance (`field-presets-streaming-preview-design.md:52-59,74`) — debounced ~150ms, ≤1 in flight, and off by default, so the exposure exists whenever the user enables the feature, which is its intended use. Full renders and TurboForge benchmarks run on the same GPU. WebGL and CUDA/DirectML contend directly.
- **How it fails:** A heavy constellation frame (bloom chain, particle sim, high-res fabric) steals GPU time precisely while the user drags a ghost — previews blow their latency budget, the drag stutters, and the app's headline interaction dies on 8GB-class hardware.
- **Mitigation:** Budget the entire overhaul at **≤4ms GPU per frame on low/mid tiers (≤6ms high) while the diffusion queue or a preview is active** — the same numbers as the Performance Budget table. Add one automatic downshift: when `queue` has a running job or a preview is in flight (both visible in the store), drop effects one quality level. This is the *useful* half of "dynamic quality scaling" — triggered by a known cause, not by noisy FPS readings.

### R3 — Instanced stars rewrite the orb pipeline for zero visible gain — **High**
- **Why it matters:** Every orb is one Mesh with its own ShaderMaterial on a shared 48×32 sphere (`scene.ts:151-170`), and that per-node material is load-bearing: in-place gradient retint (`scene.ts:173-178`), the interactive ring dial writing params back (`Graph3DView.tsx:1404-1409`), per-orb raycast picking, expand-to-DOM-card, and CSS3D chip anchoring. The documented budget — 60fps at 40 nodes — is assumed met today (no complaints on record), though nothing measures it; Phase 0's frameStats establishes the real baseline before any instancing question could be revisited.
- **How it fails:** Weeks of migration to InstancedMesh + per-instance attribute plumbing, regressions in picking/dial/expand (the "Editing never breaks" invariant, `living-constellation-vision.md:58`), and no user-visible improvement because draw calls were never the bottleneck. The actual per-frame costs today are elsewhere: ring `TorusGeometry` rebuilt per value change (`scene.ts:185-200`, unquantized in the playback path at `Graph3DView.tsx:949-957`), `routeWiresLive` doing O(E·(E+N)) `.find()` lookups plus 28-point bezier re-tessellation per wire per frame (`Graph3DView.tsx:875-905`), and CSS3D DOM transforms every animated frame.
- **Mitigation:** **Cut instanced stars.** Keep per-mesh orbs. Reserve instancing/Points for *new* layers where counts are genuinely large (particle dust). Fix the real hot paths as by-catch: quantize the playback ring path like audio already does (`Graph3DView.tsx:1128-1144`), build an id→node/edge Map for `routeWiresLive`, and parameterize the ring arc in-shader so the torus is never rebuilt.

### R4 — Hidden-window rendering gets multiplied, not throttled — **High**
- **Why it matters:** The proposal lists "hidden-tab throttling" as a feature to add; the codebase is deliberately **anti**-throttled. The playback driver arms a 30ms wall-clock timer precisely so it survives rAF starvation, and every tick unconditionally calls `renderer.render + cssRenderer.render` (`playbackClock.ts:88,120-150`; `Graph3DView.tsx:1036-1037`). A minimized window doing motion playback burns ~30Hz of GPU forever. There is zero visibility handling anywhere (`grep visibilitychange|document.hidden` = 0 hits; `src-tauri/src/main.rs` registers no window events).
- **How it fails:** Add fabric + particles + bloom to that same direct-render path and a minimized LumenDeck consumes a laptop's GPU/battery invisibly — while possibly starving a running diffusion job.
- **Mitigation:** Split **state advance** from **scene work** in the playback driver. The wall-clock playhead must keep advancing while hidden (that is the hard-won starvation invariant — keep it), but a hidden tick runs *only* the pure playhead math (`advancePlayback` + the 6Hz store write): skip `applyPlaybackFrame`, `routeWiresLive`, and both renders entirely — gating just the draw calls would leave ring-geometry rebuilds and the O(E·(E+N)) wire loops burning CPU invisibly, the same bug with the GPU half removed. On `visibilitychange` → visible, apply one frame at the current playhead and render once (which also refreshes the CSS3D DOM transforms, so no staleness case remains). Same gate for the settle clock. Existing injected-hook tests assert ticking, not pixels, so their semantics are preserved.

### R5 — Bloom: expensive, decision-reversing, and it taxes the encodings it is supposed to serve — **High**
- **Why it matters:** "No postprocessing passes (no bloom pipeline)" is an explicit, documented decision tied to the only render budget on record (`graph3d-and-image-node-design.md:62-63`). Full-screen bloom at DPR 2 in a 1440×900+ window is the single most expensive proposed item (multiple render targets + blur chains). The scene already fakes glow with additive blending + fresnel + fog, and the transparent-canvas/CSS3D sandwich means post-FX applies only to the WebGL layer, shifting its look away from the DOM cards sitting on top. Perceptually, bloom is a low-pass filter over exactly the brightness differences any luminosity encoding would use.
- **How it fails:** An UnrealBloomPass-style chain lands, the 60fps@40-nodes budget quietly dies on integrated GPUs, bright orbs smear into each other so the weight-gradient and emissive encodings become unreadable, and the neon look doubles up (additive materials + bloom = white mush).
- **Mitigation:** No bloom through Phase 4. Data-driven "glow" = a per-orb **emissive intensity uniform** in the existing orb shader plus a billboard halo sprite — one uniform, zero passes. If Phase 5 still wants true bloom, it must be: selective (threshold), ≤½-resolution, ≤2 passes, behind a flag, **default off**, with an intensity slider, measured on 8GB hardware, **alpha-preserving** (the canvas is `alpha: true` composited over DOM — stock UnrealBloomPass writes an opaque composite that would black out the DOM background behind the graph; a custom additive composite that leaves destination alpha intact is mandatory), and written up as an explicit reversal of the v0.12 decision.

### R6 — Visual-channel collision with three shipped owner systems — **High**
- **Why it matters:** The proposal wants eight new encodings, but the channels are not free. Color gradient **and** ring arc are both already bound to one scalar (primary weight) (`orbWeight.ts:150-201`); position and scale are exclusively owned by motion XOR audio with hard override-and-restore semantics — no compositor exists (`store.ts:818-821,1501-1503`; `restoreStaticOrbs`, `Graph3DView.tsx:962-989`); the audio path even temporarily hijacks the gradient tint from the ring value (`Graph3DView.tsx:1137-1142`). The brand ramp cyan→violet→mango is pinned by spec.
- **How it fails:** Temperature→color fights the weight ramp; velocity→position fights motion/audio ownership; a new layer writes scale, then `restoreStaticOrbs` stomps it on audio stop; users can no longer tell whether a violet orb means "high cfg" or "warm temperature". Eight variables into ~4 genuinely free channels is unreadable regardless of implementation quality.
- **Mitigation:** A single **encoding registry module** (pure, unit-tested) is the only place allowed to bind datum→channel, and it declares ownership per state (idle/playback/audio). New encodings use only the *unclaimed* channels: fabric depression depth, emissive intensity, halo fuzz/radius, opacity, trails, and event ripples. Color and ring stay exactly as shipped. Maximum **4 always-on encodings**; everything else is a toggled lens or tooltip. See Data Encoding Rules.

### R7 — The proposed variables mostly don't exist in the data model — **High**
- **Why it matters:** Mass exists (`primaryWeight`). Anomaly exists (`health: HealthIssue[]`, 11 codes, recomputed per commit — `core/health.ts:19-25`, `store.ts:657-658`). Everything else is missing: `WorkflowNode` has no `createdAt` (no age), no usage counters (no activity), `QueueJob` carries **no nodeId** so render events cannot be attributed to a node (`store.ts:169-182`), there are no stored embeddings or confidence values (no uncertainty/temperature), and "velocity" has no definition (orbs don't move except under motion/audio overlays). The README's Constellation System section already *promises* encodings like freshness/quality/drift (`README.md:42-148`) with no code behind them — in a repo whose doctrine is "measured-only" honesty.
- **How it fails:** With no data to bind, the effects ship as decoration with astronomy names — precisely the "pretty but useless" outcome, and a direct violation of the product's honesty rules (`turboForge.md:52-56`, fallback-honesty in `diagnostics.md`).
- **Mitigation:** **Data plumbing is Phase 0**, before any pixel changes: add `nodeId` to QueueJob at enqueue time; add a `nodeMeta` side-map (createdAt, lastRenderAt, renderCount, lastEvolveScore) as an additive persisted slice (the `field.ts` hydrate pattern) — respecting "no workflow-schema churn" (`living-constellation-vision.md:62-64`). Every encoding in this document names its exact datum; anything without one was cut.

### R8 — Stateful wave/ripple simulation vs. variable timesteps and a starvation history — **Medium**
- **Why it matters:** "Water-like" ripple propagation normally means a ping-pong wave-equation sim — state that integrates over time. This codebase's frame cadence is deliberately irregular: rAF racing 30/34ms timers, a shipped rAF-starvation bug (v0.13.1, commit 4629f29), and specs that mandate wall-clock steppers "NOT bare rAF". Stateful sims are exactly what explodes under irregular dt.
- **How it fails:** Fabric oscillates or blows up after a starve/resume; weeks of damping/tuning; a sim texture pass per frame defeats R1's sleep guarantee because state must keep integrating even when nothing changed visually.
- **Mitigation:** **Stateless, analytic fabric.** Displacement = Σ Gaussian wells (uniform array from node positions + weights) + Σ ≤8 active ripple wavefronts, each an analytically decaying expanding ring that is a pure function of `(now - eventTime)`. No sim state, no ping-pong FBOs, no dt sensitivity — starvation-immune by construction, unit-testable as pure math on the CPU mirror, and it sleeps perfectly (a settled fabric is just a fixed function of well positions). True interfering wave propagation is deferred to Phase 5 and, frankly, expected to be rejected there.

### R9 — Maintainability: the overhaul lands in a 1,716-line component with three render paths — **Medium**
- **Why it matters:** `Graph3DView.tsx` already hosts six rAF users, five reconcile effects, three direct-render bypass paths (playback `:1036`, audio `:1161`, scrub `:1093`), and manual override/restore choreography. The store is ~114KB. There is no centralized per-frame hook: a new animated layer today must patch all three loops *and* the idle flush.
- **How it fails:** Five new subsystems threaded through this become an unmodifiable uber-component; the next feature after the overhaul takes 3× as long; subtle ordering bugs between overlays multiply.
- **Mitigation:** Follow the codebase's own overlay contract ("own groups… added/removed alongside, not within", `Graph3DView.tsx:449-454`): each layer = a pure module in `graph3d/` (the `scene.ts`/`orbWeight.ts` convention: no DOM, unit-tested) + its own THREE.Group + one gated effect. Add a tiny **frame-hook registry** (`onFrame(cb)` called from the flush body and both continuous loops) so layers subscribe once instead of patching three loops. Refuse any phase that grows `Graph3DView.tsx` by more than ~150 lines.

### R10 — No instrumentation and no render tests: every claim in this overhaul is currently unverifiable — **Medium**
- **Why it matters:** There is no fps counter, frame-time metric, or `renderer.info` readout anywhere; `core/diagnostics.ts` reports backend/storage only. Vitest runs in `environment: 'node'` (`vite.config.ts:76-79`) — the entire THREE scene graph, shaders, and every loop are untested and untestable in the current harness. The house rule is "measured-only speed language".
- **How it fails:** Quality tiers ship with made-up thresholds; a regression that halves frame rate on iGPUs is invisible until a user complains; "60fps at 40 nodes" silently stops being true.
- **Mitigation:** Phase 0 ships a pure `frameStats` module (EMA frame time, worst-1%, draw-call count via `renderer.info`) surfaced on the Diagnostics page, plus unit tests for all new pure modules (fabric math, encoding registry, tier governor, settle clock — with injected clocks, matching the `flushScheduler`/`playbackClock` test style). One minimal end-to-end smoke (dev-server + Playwright, or tauri-driver) that mounts the real 3D view, loads a seeded workflow, and screenshots at fixed camera — the first render-path test the project has.

**Honorable mentions (folded into phases):** no `webglcontextlost/restored` recovery — a mid-session GPU reset (likelier under higher GPU load) currently leaves a dead canvas (`Graph3DView.tsx:419-426` handles creation only) → Phase 0 fix. `devicePixelRatio` captured once, wrong after moving to a different-DPI monitor (`Graph3DView.tsx:433`) → Phase 0 fix. `prefers-reduced-motion` read once with no change listener and ignored inside the 3D view (`GraphWorkspace.tsx:28-52`) → Phase 4 fix. MSI ≤16MB gate — no texture/asset payloads; everything procedural in GLSL.

## Performance Budget

Context that shapes every number: desktop Tauri (WebView2/Chromium, ANGLE→D3D11 on Windows), transparent canvas over DOM, DPR ≤2, one window ~1440×900 default, **and a diffusion renderer sharing the GPU**. The only inherited budget is 60fps at 40 nodes with no post-processing. Frame-time budgets below are for the WebGL+CSS3D work only, measured via the Phase 0 `frameStats` EMA.

| Budget | Low (Intel iGPU / shared 4–8GB) | Mid (8GB dGPU: GTX 1650–RTX 3050, Apple M1) | High (RTX 3060+ / M2 Pro+) |
|---|---|---|---|
| Frame time (active states) | ≤ 12ms (target 30–60fps, floor 30) | ≤ 8ms (target 60fps) | ≤ 6ms (60fps locked, DPR 2) |
| Frame time while diffusion queue/preview busy | ≤ 4ms (drops one quality level) | ≤ 4ms | ≤ 6ms |
| Idle (settled) | **0 draws** — sleep invariant, all tiers | 0 | 0 |
| Draw calls (60-node graph)¹ | ≤ 200 | ≤ 280 | ≤ 380 |
| Particle count (dust layer) | 0 (off) | ≤ 1,000 | ≤ 2,000 |
| Gravity "field"² | 64-well uniform array (no texture) | 64-well uniform array | 64-well uniform array |
| Fabric mesh density | 64×64 plane (~8k tris) | 128×128 (~33k tris) | 192×192 (~74k tris) |
| Bloom passes | 0 | 0 (Phase 5 flag: 0 default) | 0 default; Phase 5 flag: threshold + ≤2 blur passes at ≤½ res, ≤2.5ms measured |
| Added GPU memory (whole overhaul) | ≤ 16MB | ≤ 32MB | ≤ 64MB |
| CPU/GPU sync | **Zero readbacks, all tiers.** No `readPixels`, ever. Picking stays CPU raycast. Uniform/attribute uploads ≤ 64KB/frame. | | |
| Minimum acceptable FPS | 30 during playback/audio | 55 | 60 |
| DPR | 1 (drop from 2) | min(devicePixelRatio, 2) | min(devicePixelRatio, 2) |

¹ Draw-call caps are provisional: the pre-overhaul base scene alone is ~150–180 calls at 60 connected nodes (per-mesh orbs + rings + per-edge wires + grids). Phase 0's `renderer.info` snapshot recalibrates these caps from measurement. Every *new* layer is batched — halos ship as one `Points`/instanced layer (single draw, all tiers), dust one draw, fabric one draw — so the overhaul adds ≤4 draw calls total at any tier.

² More than 64 nodes: clamp to the 64 deepest wells + a diagnostics warning, **all tiers**. The R16F field-texture path is recorded as a future escape hatch, out of scope for these phases.

Tier selection: the hardware column is probed once at 3D-view mount (GPU renderer string via `WEBGL_debug_renderer_info`, device memory hints) and maps to a default of the single user-visible quality setting `graph3dEffects: 'off' | 'minimal' | 'standard' | 'rich'` — iGPU → `minimal`, 8GB-class dGPU → `standard`, high-end → `rich`. **No continuous FPS-based scaling.** Exactly two automatic downshifts, both cause-driven, both defined on the quality enum (`rich → standard → minimal → off`): (1) diffusion queue/preview active → one quality level down for the duration; (2) frame EMA > 33ms sustained for 5s → one level down for the session, with a toast, never back up automatically. Manual setting always wins.

## Rendering Architecture Recommendation

The governing principles: **stateless over simulated** (R8), **uniforms over textures** at this entity count, **zero new render targets through Phase 4**, **no new loops** — only new subscribers to existing ones plus a settle clock.

| Stage | Runs | Details |
|---|---|---|
| **CPU (per store commit, not per frame)** | Encoding registry: derive per-node visual params (well strength from `weightT`, emissive from `nodeMeta`, halo from health/fallback) → write uniforms/attributes. Well *rest* positions from `orbWorldCenter`. Event queue for ripples (timestamped on push). | This is the existing reconcile-effect pattern; nothing here runs when nothing changes. During motion/audio overlays — which move orbs via direct scene mutation, no store commits — well xz re-packs from live `group.position` inside the frame hooks (a ≤1KB uniform upload), so depressions track their orbs instead of visibly detaching. Phase 2+; Phase 1 documents wells-at-home as the interim behavior. |
| **CPU (per active frame)** | Particle integration (≤2k): Euler drift sampling a **precomputed coarse gradient grid** (64×64 Float32 vec2, rebuilt only when wells change — same trigger as the uniform pack), bilinear-sampled per particle so cost is O(1) per particle, not O(wells); write one dynamic BufferAttribute. Trail ring-buffers; settle-clock bookkeeping. | The naive version — 2k particles × 64 analytic `exp` gradients = ~128k `Math.exp`/frame ≈ 1.5–3.5ms of main-thread JS — blows the budget; the grid mechanism is what makes the stated ≤0.5ms at 2k particles honest. |
| **Vertex shader** | Fabric plane displacement: `y = -Σᵢ dᵢ·exp(-r²/2σᵢ²) + Σⱼ ripple(t-tⱼ, r)` over ≤64 wells (uniform vec4 array: xz, depth, σ) and ≤8 ripple events (uniform vec4: xz, t₀, amp). Orb shader untouched except new uniforms. | Pure function of uniforms + time — starvation-immune, testable via a CPU mirror of the same math. WebGL2 guarantees ≥256 vec4 vertex uniforms; 64 wells + 8 ripples fits with huge margin. |
| **Fragment shader** | Fabric shading: depth-tinted brand-token gradient, equipotential contour lines via `fwidth` (standard derivatives — core in WebGL2), and a distance fade implemented as **alpha→0** (premultiplied-consistent) — *not* scene fog. Orb: existing gradient/fresnel + `uEmissive` term; halos batched in one `Points` layer. | The canvas is `alpha: true` over DOM: THREE.Fog lerps RGB toward opaque `#071426` without touching alpha, so a fog-faded filled plane becomes a solid slab over the DOM background (and custom ShaderMaterials don't inherit fog anyway). Fabric material: `fog: false`, `depthWrite: false`, explicit `renderOrder: -1` — drawn beneath the additive grid/wire/ring pass so it can never stomp their transparency sort. |
| **Compute / equivalent** | **None.** No transform feedback, no ping-pong FBOs, no GPGPU. | Nothing at 60 nodes + 2k particles justifies the machinery or its debugging surface. Re-evaluate only if a future feature genuinely renders ≥50k dynamic entities. |
| **Post-processing** | **None through Phase 4.** Phase 5 optional: selective threshold bloom, ≤½ res, ≤2 passes, default off. | The transparent-canvas + CSS3D sandwich means post-FX can subtly detach the WebGL look from DOM cards — another reason bloom stays a flagged experiment. |

**Frame-hook registry (the one structural addition):** a tiny `frameHooks` module — `register(cb): unregister`, invoked with `(now, dt)` from the flush body (`Graph3DView.tsx:317-322`), the playback step, and the audio tick. Layers (fabric time, particles, trails) subscribe once; the three loops stop being individually patched. **Contract:** a hook returns `needsRender: boolean`, consumed by whichever loop invoked it. The continuous loops (playback/audio) already render every frame, so they ignore it; only the idle path translates pending decay into `requestRender()` scheduling. The settle clock therefore self-arms **only when no continuous driver owns the frame** — otherwise a decaying ripple during playback would arm the flushScheduler *on top of* the playback render and double the GPU cost per display frame, exactly when the R2 budget matters most (the codebase already carries this hazard between the audio tick and the scheduler). "Exactly one render per frame under playback + decaying ripple" is a unit-tested invariant; the settle clock reaching sleep preserves the idle invariant mechanically.

**Fallbacks:**
- Feature flag off (default at first): zero new code executes; scene is byte-identical to v0.19.1.
- WebGL create-fail / reduced-motion default / context-lost: existing 2D fallback path unchanged; **new:** `webglcontextlost`/`restored` listeners (Phase 0) that pause hooks and rebuild the scene on restore instead of today's dead canvas.
- Low tier: fabric 64×64, no particles, no ripples — still shows wells + contours (the data survives every cut).
- If the fabric flag is on but the GPU probe fails mid-flight: fabric group removed, GridHelpers restored, one loud console note — no silent degradation (house honesty rule).

## Data Encoding Rules

Strict registry — one module (`graph3d/encodings.ts`), pure, unit-tested, seeded in Phase 1 with the single mass→well entry and grown in Phase 2; **no visual layer may read store data directly.** A channel has exactly one owner per state. The proposal's astronomy vocabulary is mapped to real product data or explicitly retired:

| Proposal term | Product datum (exact source) | Visual channel | Rule |
|---|---|---|---|
| **Mass** | `weightT(primaryWeight(kind, params))` — exists (`orbWeight.ts:90-154`) | Fabric well depth + σ (NEW); existing gradient + ring arc (unchanged) | The one variable allowed redundant encoding (fabric + gradient + ring) — it is the product's core "how strong is this node" scalar. Weightless kinds (null weightT) create **no well** — flat fabric under them is the honest rendering. |
| **Velocity** | Param-change rate + overlay displacement (motion/audio/ghost drags) — derived, exists only in active states | Trail length/opacity (Phase 3) | No motion → no trail. Never encoded in color or position offset (position is owned by motion/audio overlays). |
| **Luminosity** | Activity recency: `nodeMeta.lastRenderAt` decay, `renderCount` (Phase 0 data) | Orb `uEmissive` + halo brightness (Phase 2) | Session-decaying. Bloom, if it ever exists, derives from `uEmissive` only — never an independent knob. |
| **Temperature** | — none exists | — | **Retired.** No second hue axis on a brand-pinned ramp. |
| **Uncertainty** | Inert params for current model (field-presets spec helper) + last render `fallback`/`renderMode !== 'real'` (`store.ts:158-160`) | Halo fuzz radius + dashed ring segment (Phase 2) | Always paired with the chip's text hint — never a lone visual. |
| **Age** | `nodeMeta.createdAt` (Phase 0 data) | Tooltip/chip only, for now | Deferred as a visual channel; data plumbed so a future "lens" toggle is cheap. |
| **Activity** | Render events attributed via `QueueJob.nodeId` (Phase 0 schema addition) | Ripple ping on completion (Phase 2) + feeds luminosity decay | Event bursts share the global flash limiter (≤3 onsets per rolling second) — photosensitivity + honesty (10 queued renders ≠ 10× importance). |
| **Anomaly** | `health: HealthIssue[]` — exists, 11 codes, error/warning (`core/health.ts`) | Palette-breaking ring outline (warn amber / error red), one ripple on new error, existing chip icon+text | The **only** palette-breaking channel. If everything glows red, nothing does: cap simultaneous anomaly ripples at 3, rest queue silently. |

Hard rules: (1) ≤4 always-on encodings — mass, luminosity, uncertainty, anomaly; all else behind toggles or tooltips. (2) No effect merges without a registry entry naming its datum — enforced by a unit test that walks the registry against the layer list. (3) Every animated encoding has a static equivalent (reduced-motion parity): ripple→brief ring highlight persists 2s as static; trail→direction tick; pulse→emissive step. (4) A "What am I seeing?" legend popover ships with Phase 2 — an encoding users can't decode is decoration with extra steps.

## Accessibility / Safety Requirements

- **Reduced motion:** honored *inside* the 3D view for the first time (today it only flips the 2D default, `GraphWorkspace.tsx:28-52`, and is read once with no change listener — fix both). Under reduce: fabric time frozen (wells + contours fully intact — the data story survives motionless), no ripples/particles/trails/emissive pulsing; playback still works when explicitly pressed (existing policy). Listen to `matchMedia.onchange`; an explicit in-app effects setting overrides OS in either direction.
- **Reduced bloom/flash:** bloom is default-off everywhere; if enabled (Phase 5), an intensity slider including 0. Ripple luminance delta capped (≤20% local, never full-field); no full-viewport luminance oscillation at any frequency.
- **Colorblind safety:** no new hue axes are introduced (the audit killed temperature-color), so the existing brand-ramp exposure is unchanged; anomaly = outline + icon + text per the standing "never color alone" rule; contour lines are luminance-based, not hue-based.
- **Photosensitivity:** one global flash limiter — a sliding window allowing at most **3 ripple/flash onsets per rolling 1000ms** across all systems combined (WCAG 2.3.1); anomaly and completion ripples share this single rate-limited queue. (A fixed ≥300ms spacing would permit 4 onsets in a second — the sliding window is the spec, everywhere this document mentions coalescing.)
- **User controls:** one settings surface (AppSettings optional-additive fields): `graph3dEffects: 'off'|'minimal'|'standard'|'rich'` + per-effect toggles (fabric, ripples, dust, trails) + (Phase 5) bloom slider. Persisted via the sanitize pattern; keyboard reachable; defaults conservative.
- **Pause/slow:** effects animate only in active states and settle to sleep — pausing the transport already pauses the world; no separate simulation to pause (a benefit of statelessness). For *slow*: an **effects-speed multiplier** (0.25×–1×) next to the per-effect toggles — analytic effects are pure functions of time, so slowing them is literally `t × speed`; cheap to offer, valuable for motion-sensitive users who want the encodings without the tempo.
- **Hidden window:** state advances (wall-clock invariant preserved), draws gated on visibility (R4). On restore: one clean frame, no catch-up animation burst (analytic effects are pure functions of *now* — they cannot "catch up" wrongly).

## Implementation Plan

Standing gates for every phase: full suite green (494+ and growing), `tsc --noEmit` clean, MSI builds ≤16MB, idle-sleep smoke passes, and with `graph3dEffects: 'off'` a static scene renders identically to v0.19.1 under an unchanged environment — with a **named exemption list** for the flag-independent fixes this plan deliberately ships (context-restore, DPR re-read, playback ringT quantization, hidden-window scene-work gate), each carrying its own acceptance criterion in its phase rather than hiding under a blanket byte-identical claim. Rollback for every phase: flags default off; revert = flip flag (instant) then revert commits (clean, layers are additive modules).

## Codebase Inspection Tasks — Answers

1. **Rendering framework?** Raw Three.js `^0.185.1` (`WebGLRenderer` + `CSS3DRenderer`), no react-three-fiber, no helper frameworks. React 18 owns lifecycle; Zustand 5 owns state.
2. **WebGL/WebGPU/Canvas?** WebGL2 (probe: webgl2-then-webgl, `GraphWorkspace.tsx:9-26`) inside Tauri v2's WebView2 (Chromium/ANGLE→D3D11 on Windows). No WebGPU usage or flags anywhere (`tauri.conf.json` has no browser args); the 2D editor is DOM/CSS, not canvas.
3. **Render loop?** Dirty-flag: `graph3d/flushScheduler.ts` (rAF racing a 34ms timer per invalidation, provably idle between) driving `renderer.render + cssRenderer.render` (`Graph3DView.tsx:308-333`). Three bypass paths render directly: motion playback (`createPlaybackDriver`, rAF racing 30ms wall-clock timer — **keeps rendering while hidden**, `Graph3DView.tsx:1036-1037`), audio reactivity (bare rAF, `:1150-1171`), paused-scrub store subscription (`:1082-1097`).
4. **Where are stars/entities stored?** `workflow.nodes: WorkflowNode[]` (`{id, kind, x, y, params}`) in the single Zustand store (`useStudio`, `store.ts`); reconciled into ref-held Maps of THREE Groups (one Mesh + own ShaderMaterial per orb, shared 48×32 sphere geometry) plus CSS3D DOM cards/chips.
5. **Where are visual variables defined?** `graph3d/orbWeight.ts` (primaryWeight/weightT → brand-ramp `gradientStops` + `ringAngle`), `graph3d/scene.ts` (the only custom GLSL: orb gradient/half-lambert/fresnel; wire/ring/grid builders), brand tokens in CSS resolved via `resolveCssColor`. Overlay drivers: `core/motion/*`, `core/audio/*`, `core/field/*`.
6. **Settings system?** `state/appSettings.ts` — flat sanitized interface persisted in the `lumendeck.v1` localStorage blob; established optional-additive-field pattern (`graphMode`, `graph3dStyle`) is the feature-flag precedent. No dedicated flag system.
7. **Performance monitor?** None. No fps/frame-time instrumentation anywhere; `core/diagnostics.ts` covers backend/queue/storage only. (Phase 0 fixes this first.)
8. **Existing shaders?** One pair: `ORB_VERTEX_SHADER`/`ORB_FRAGMENT_SHADER` (`scene.ts:109-148`). Everything else is stock materials with additive blending.
9. **Existing post-processing?** None — explicitly rejected in the v0.12 spec ("no bloom pipeline", budget 60fps@40 nodes). Glow is faked via additive blending, fresnel, fog, and CSS.
10. **Safest first file to modify?** `state/appSettings.ts` (add the optional flag — zero render risk), then a **new** pure module in `graph3d/` mounted as its own Group — the codebase's documented overlay contract (`Graph3DView.tsx:449-454`: "added/removed alongside, not within"). `Graph3DView.tsx` is touched last and minimally.
